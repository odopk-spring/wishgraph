# WishGraph 方法：在两个黑箱之间建立项目接口

[English](wishgraph-method.en.md) | [简体中文](wishgraph-method.md)

## 1. 为什么模型越强，项目反而越难理解

AI 可以在几分钟里修改上千行代码，但人类理解改动、检查影响和确认项目状态的速度没有同步提高。

一个小要求可能从页面动画牵到网络状态、缓存、导航和持久化。每一处修改单独看都很合理，几轮之后却很难回答：

- 这次到底改了哪些行为？
- 有没有影响其他模块？
- 之前确认的约束还有效吗？
- 新窗口或新模型知道项目已经走到哪里吗？

模型本身难以完全预测；当项目的大部分代码也由 AI 生成时，项目结构、历史决定和隐含约束对人类同样可能变成黑箱。让一个无法完全预测的系统直接管理另一个越来越难理解的系统，失控就不只是偶然错误，而是流程结构的问题。

## 2. 自然语言和代码之间缺少一层接口

人类说的是体验：“动画自然一点”“这个颜色不好看”“不要让用户看到加载闪烁”。Agent 接触到的却是整个代码系统。

自然语言天然低带宽，也经常模糊、变化或前后矛盾。问题不在于不能用自然语言开发，而在于不能让未经整理的自然语言直接获得修改整个项目的权限。

WishGraph 增加一层项目接口：

```text
Wish → Requirements → Spec → Task → Worker → Validation → Report → Current State
```

它不要求用户先学会写工程规格。Discussion 负责把愿望整理成可以执行、验证和回滚的 Task；Worker 只执行这个 Task；Integration 再把真实结果写回项目状态。

## 3. 压缩项目规则，而不是反复背答案

如果 Agent 每次进入项目都重新翻聊天记录、扫描代码、猜测上次为什么这样改，它只是在背一张越来越长的答案表。

更有效的方式是抽出项目里的规则：

- 项目解决什么问题。
- 系统由哪些模块组成。
- 模块怎样连接，哪份数据是真相源。
- 这次允许改什么，明确不改什么。
- 怎样证明修改完成。
- 修改后哪些项目事实必须同步。

这些规则比完整聊天和完整源码更短，却能支持后续推理。复杂项目并没有因此变简单，但 Agent 不必每次从第一行重新理解它。

WishGraph 把这种压缩落到几类项目事实：

- **Spec Graph：**项目应该是什么。
- **Dependency Map：**功能、模块、文件、合约和验证怎样连接。
- **Task Graph：**当前工作怎样拆成有边界、可排序、可回滚的执行单元。
- **Causal Log：**项目为什么变成现在这样，包括决定、报告、失败和修复证据。
- **Current State：**人类和新 Agent 最快需要看到的当前快照。

## 4. 一个不断闭环的工作循环

WishGraph 的普通循环很短：

```text
讨论 → 授权 → 执行 → 验证 → 集成 → 呈现结果 → 继续讨论
```

### Discussion：把想法说清楚

用户仍然使用普通自然语言。Discussion 结合当前项目状态追问少量关键问题，把需求整理成明确 Task，并说明范围、非目标和验证方法。

Discussion 不修改业务代码。它的职责是决定“下一步应该做什么”，而不是在同一个上下文里一边讨论、一边实现、一边给自己的结果做评审。

### Worker：按边界执行

Worker 位于独立、用户可查看和控制的 Agent thread 或窗口。它只读取准确 Task、执行规则、必要状态和范围内源码。`enforce` 要求修改前取得绑定 Claim；`warn` 不因 Claim 自动化缺失而阻塞。

完成时，Worker 留下一份不可变 Run Report，说明修改、验证、风险和需要更新的项目事实。自然语言“完成了”不能替代这些证据。

### Integration：让项目记住变化

有运行时自动化时，Worker 终态进入 `integration_pending`；否则 `warn` Worker 直接返回报告和结果 commit。安全结果由 Discussion-local Integration 检查、合并和回写。

`reports/PROJECT_STATUS.md` 保持为最新快照，历史留在 Run Reports 和 Git 中。下一个窗口从当前状态继续，不需要复制完整提示词或重新介绍项目。

## 5. 人类并没有退出流程

WishGraph 不是让 AI 接管所有判断。人类仍然负责：

- 提出需求和不满。
- 判断产品方向与取舍。
- 授权执行准确 Task。
- 接受风险和重大变化。
- 评审最终结果。

变化的是，人类不再需要每次把模糊愿望手工翻译成几十条工程命令，也不需要在窗口之间搬运全部背景。系统把需要人类判断的内容压缩成少量具体问题，把机械执行和证据收集交给 Agent。

Review 因此是一种 Discussion 中的呈现状态，不是第四个 Agent。用户看到的应该是：系统如何理解需求、准备做什么、实际改了什么、怎样验证、还剩什么风险，以及下一步建议。

## 6. 反黑箱不等于假装完全可解释

WishGraph 不承诺解释模型内部，也不承诺让任何复杂项目瞬间变得简单。它做的是改进黑箱周围的输入、输出和控制边界。

每次执行至少应该回答：

- 它理解的人类意图是什么？
- 这次 Task 的范围和非目标是什么？
- 为什么修改这些文件？
- 运行了什么验证？
- 哪些风险仍然存在？
- 项目目标、架构、代码地图或当前状态是否需要更新？

这不是为了增加流程文件，而是为了让人类能够评审，让后续 Agent 能够接手，也让错误可以追到具体假设。

## 7. 调试要追因果，不要猜熟悉文件

常见修复路径是看到页面出错，就去熟悉的 View 里试一个补丁。WishGraph 更关注因果链：

```text
Error → State → Code → Spec
```

先确认现象，再找到最早出现问题的状态，定位写入或读取这份状态的代码，最后回到规格判断预期行为。如果规格本身不清楚，问题应该先回到 Discussion，而不是让 Worker用更大的 patch 掩盖歧义。

目标始终是最小、可验证、可回滚的修改集合。

## 8. WishGraph 和 Harness Engineering

WishGraph 属于广义的 Harness Engineering：它组织 Agent 可以读取的项目知识、执行边界、反馈循环和机械验证。

它选择把**长期项目连续性**放在中心：换窗口、换模型、换 Agent 或换宿主后，项目的目标、结构、任务、证据和当前状态还能不能保持清楚；人类还能不能理解为什么走到这里，以及下一步为什么这样安排。

因此，WishGraph 更适合作为一种具体实践，而不是所有 Agent 工程问题的总方案。它不替代版本控制、测试、代码审查、权限系统、容器或操作系统沙箱；它把这些能力组织进一条可以持续交接的项目循环。

## 9. 更大的方向

Agent 正在把计算机的控制入口继续向自然语言上移。模型越能快速完成局部任务，项目越需要清晰的状态、可靠的记忆和可审计的控制回路。

WishGraph 做的是一种压缩：把庞大、混乱、不断变化的项目压缩成一套人类和 AI 都能继续理解、推理、交接和验证的结构。项目没有变小，但它重新变得可以驾驭。
