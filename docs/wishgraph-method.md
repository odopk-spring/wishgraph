# 许愿式编程：WishGraph 方法论

## 一句话
提出需求-编译执行规格-执行并报告——提出新的需求......

构建一种由人类低带宽意图驱动、由 AI 自动编译需求、维护结构、执行修改、追踪因果、自我修复，并通过简化报告接受人类评审的复杂项目自治工程系统。

人类一句话能传达的信息太少，那就让ai把传达的信息扩展成执行规范，再让另一个ai去只阅读执行规范再去执行。为了让这个系统稳定，我们把项目架构沉淀成外置的数
据库，解放单一会话窗口的注意力资源。不同窗口阅读共同的项目进度，你可以在任何窗口、任何agent之间使用配套好的提示词无缝衔接。
claude code讨论、codex或deepseek执行的工作流在新的项目框架下将变得无比丝滑。

## 1. 人类从执行者变成评审者

传统开发要求人类知道怎么拆需求、改哪些文件的哪几行代码，同时还不影响全局。

WishGraph 里，人类主要做两件事：

- 表达意图：这个页面更流畅一点、聊天像真人一点、不要太复杂、这个结果我不喜欢。
或者换个更诗意的表达，人类只负责“扇动蝴蝶的翅膀”，给AI+复杂项目架构的自洽系统提供扰动。

- 做评价：通过、不对、方向偏了、继续优化、这个风险我接受。或者随时更新你的想法给项目文档加上新需求。

人类不再负责管理所有实现细节，而是负责方向、偏好、风险和最终判断。

哪怕是vibe codeing也不需要每隔几个对话窗口就让ai帮你总结整理目前的项目架构。

## 2. AI 从代码助手变成项目操作系统-把决策窗口和执行窗口分开。

普通 Agent 是：

```text
自然语言 -> 执行一个任务 -> 返回结果
```

WishGraph 是：

```text
自然语言 -> Spec -> Task -> Code -> Verify -> Repair -> Log -> Report -> Human Review
```

AI 同时承担需求工程师、架构师、执行者、测试者、调试器、文档维护者和项目汇报者的工作。

“一生二、二生三、三生万物”。你负责提供1，让ai去生成更复杂的万物。
关键不是让 AI 一步从 1 到万物，而是把系统拆成 AI 能稳定完成的许多小跳跃。

## 3. 用外部项目结构替代聊天上下文

AI 的上下文窗口有限。复杂项目真正可靠的记忆不该放在聊天里，而应该放在项目文件系统里。

WishGraph 的外置记忆包括：

- Spec Graph：项目应该是什么。
- Dependency Map：模块和功能如何依赖。
- Causal Log：为什么变成现在这样。
- Probe：如何自检和发现退化。
- Review Window：人类需要看的压缩状态。

换会话、换模型、换 Agent 后，只要读取这些文件，就能继续工作。

## 4. 反黑箱原则

AI本身是一个黑箱而你不理解的项目文件夹本身对你而言也是一个黑箱。

黑箱问题的本质是解耦黑箱在系统的作用，且最大限度的优化它的输入输出。

前面已经回答过了，把讨论窗口和执行窗口分开，不要让ai这个黑箱直接去管理你的项目（更大的黑箱）

使用良好的执行规范，确保ai能理解你的意图，修改代码同时不影响其他板块。


AI 每次执行都必须能回答：

- 它理解的人类意图是什么。
- 它把意图编译成了哪些规范。
- 它为什么只改这些地方。
- 有什么风险。
- 怎么验证。
- 结果是否通过。
- 如果失败，错误从哪里传导过来。

这不是为了形式主义，而是为了让人类能评审，让后续 Agent 能接手。

## 5. 错误处理：追因果链，而不是猜文件

传统调试容易变成：

```text
页面错了 -> 去熟悉的 View 文件里试补丁
```

WishGraph 要求：

```text
Error -> State -> Code -> Spec
```

先看错误现象，再找错误状态，再看哪个代码路径写入或读取了它，最后回到规范判断行为是否应该如此。

目标是 minimal patch set，而不是大改一通。

## 6. 理想的人类评审界面

理想产品不是让人类在多个窗口复制粘贴，而是只面对一个评审窗口。

它展示：

- AI 对需求的理解。
- 即将执行的计划。
- 执行后的摘要。
- 风险与验证结果。
- 需要人类判断的选择。

背后可以有很多 Agent、任务、日志和代码修改，但人类看到的是可理解的项目状态摘要。

## 7. 更大的判断

计算机抽象层一直在上移：

```text
二进制 -> 汇编 -> C -> 高级语言 -> GUI -> 自然语言 Agent
```

新层不会消灭旧层，而是把旧层压到下面。

WishGraph 不是最终形态，但它描述了下一层工程接口：人类向系统用自然语言提出需求，AI 维护一个可解释、可修复、可成长的软件世界。
